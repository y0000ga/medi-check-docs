# Medi Check 產品規格書
Product Requirements Document｜版本 1.1｜2025

---

## 1. 產品概述

### 1.1 產品定位

Medi Check 是一款行動端藥品提醒與服藥管理應用程式，協助家庭照護者與病人本人建立規律的服藥習慣、追蹤服藥歷史，並支援多人共同照護同一位病人。

### 1.2 目標用戶

- **家庭照護者**：需要替家人或寵物管理藥品與提醒排程的用戶
- **病人本人**：自行管理個人藥品與服藥紀錄的用戶

### 1.3 核心解決問題

- 忘記服藥或服藥時間不規律
- 多位照護者需共同管理同一病人的藥品資訊
- 缺乏結構化的歷史服藥紀錄

### 1.4 產品範圍

**In Scope**
- 使用者帳號管理（註冊、登入、個人資料編輯）
- 病人資料管理
- 藥品資料管理
- 提醒排程設定與管理
- 服藥紀錄追蹤與編輯
- 照護關係建立（邀請機制）與查看

**Out of Scope**
- 離線使用（規劃為未來功能）
- 多語言支援（規劃為未來功能）
- 藥品資料庫整合（如健保資料庫）
- 推播通知系統（本文件不含後端通知實作細節）

## 2. 使用者角色與權限

### 2.1 帳號與病人的關係

每個帳號在註冊後，系統會自動為其建立一個對應的「自身病人」紀錄，代表用戶本人（或用戶主要管理的對象，如寵物）。用戶可在此基礎上額外建立或透過邀請關聯更多病人。

### 2.2 角色定義

| 角色 | 說明 |
|------|------|
| 照護者（Caregiver） | 對特定病人擁有管理權限的用戶。可透過邀請機制取得對其他帳號所屬病人的存取權。 |
| 被照護者（Patient） | 病人實體，可對應至另一個系統帳號，或純粹作為資料紀錄（如寵物）。 |

### 2.3 照護關係說明

病人（Patient）有兩種建立方式：

- **由照護者直接建立**：不對應任何系統帳號，例如寵物或無法自行操作的被照護者
- **對應至另一個帳號**：被照護者本人擁有系統帳號，透過邀請流程與照護者建立關聯

### 2.4 照護關係權限等級

| 權限等級 | 可執行操作 |
|----------|-----------|
| 讀取（Read） | 可查看病人的藥品、提醒排程、服藥紀錄 |
| 寫入（Write） | 可新增、編輯藥品、排程與服藥紀錄 |
| 管理員（Admin） | 包含寫入權限，另可管理照護關係（邀請、撤銷） |

## 3. 功能規格

### 3.1 登入與註冊

#### 3.1.1 功能描述

使用者可透過註冊頁建立帳號，登入後取得 Token 以存取受保護的頁面與後端 API。

#### 3.1.2 使用情境

- 新用戶透過表單填寫基本資料完成註冊
- 已有帳號的用戶輸入憑證後登入，系統回傳 Token 並導向首頁
- Token 過期後需重新登入

#### 3.1.3 欄位定義與驗證規則

**登入欄位**

| 欄位 | 說明 |
|------|------|
| 電子郵件 | 必填，格式須符合 email 規範，前端即時驗證 |
| 密碼 | 必填，6–18 個字元，須包含大寫字母與特殊字元（!@#$%^&*），不允許空白字元 |

**註冊欄位**

| 欄位 | 說明 |
|------|------|
| 使用者名稱 | 必填 |
| 電子郵件 | 必填，格式須符合 email 規範，前端即時驗證 |
| 生日 | 必填，不可晚於今日 |
| 密碼 | 必填，6–18 個字元，須包含大寫字母與特殊字元（!@#$%^&*），不允許空白字元 |
| 確認密碼 | 必填，須與密碼欄位完全一致 |

#### 3.1.4 頁面狀態

| 狀態 | 行為 |
|------|------|
| 表單驗證錯誤 | 前端即時判斷，欄位下方顯示對應錯誤訊息（email 格式錯誤、密碼不符規則等） |
| 操作失敗 | 後端回傳錯誤後顯示於表單頂部，與前端表單驗證分開處理 |

---

### 3.2 首頁與提醒事件

#### 3.2.1 功能描述

首頁為用戶登入後的主畫面，集中顯示當日待服藥的提醒事件，協助用戶快速掌握服藥狀況。

#### 3.2.2 顯示邏輯

- 依當前登入者的可見病人範圍顯示相關提醒
- 顯示當日需服用的藥品與時段，依排程時間升冪排列
- 不同照護關係下可見的病人資料不同
- 用戶可直接在首頁點擊「已服用」執行快速確認（quickCheck）

#### 3.2.3 頁面狀態

| 狀態 | 行為 |
|------|------|
| 載入中 | 顯示載入指示器 |
| 空白狀態 | 顯示「今日沒有服藥事件」文字說明，提供「新增排程」捷徑 |
| 正常狀態 | 顯示當日提醒事件列表，支援下拉重新整理 |
| 錯誤狀態 | 顯示錯誤訊息與重新整理按鈕 |

---

### 3.3 病人管理

#### 3.3.1 功能描述

用戶可管理其照護的病人列表，包含新增、編輯與查看病人基本資料。病人資料為藥品、提醒排程與服藥紀錄的關聯基礎。

#### 3.3.2 欄位定義

| 欄位 | 說明 |
|------|------|
| 病人名稱 | 必填，識別病人用 |
| 生日 | 必填，不可晚於今日 |
| 筆記 | 選填，可描述病人特殊情況或照護需求 |

#### 3.3.3 使用情境

- 查看所有可存取的病人列表（支援關鍵字搜尋、排序）
- 新增一位新病人（直接建立，不對應帳號）
- 編輯現有病人基本資料

#### 3.3.4 頁面狀態

| 狀態 | 行為 |
|------|------|
| 載入中 | 顯示載入指示器 |
| 空白狀態 | 顯示病人列表為空的說明文字 |
| 正常狀態 | 顯示病人列表，支援搜尋、排序與分頁 |
| 錯誤狀態 | 顯示錯誤訊息與重新整理按鈕 |

---

### 3.4 藥品管理

#### 3.4.1 功能描述

用戶可管理特定病人的藥品資料，支援列表瀏覽、搜尋、篩選與新增、編輯流程。

#### 3.4.2 欄位定義

| 欄位 | 說明 |
|------|------|
| 所屬病人 | 必填，關聯至病人實體 |
| 藥品名稱 | 必填，藥品的識別名稱 |
| 劑型 | 必填，可選項目（錠劑、膠囊、軟膠囊、藥丸、液體、糖漿、懸浮液、滴劑、粉末、顆粒、藥包、注射劑、小瓶針劑、安瓿、吸入劑、噴劑、霧化液、乳膏、軟膏、凝膠、乳液、貼片、栓劑、眼藥水、耳滴劑、鼻噴劑、其他） |
| 筆記 | 選填，備註用途（如服用注意事項） |

#### 3.4.3 列表功能

- 支援關鍵字搜尋（藥品名稱）
- 支援劑型篩選
- 支援排序與分頁
- 列表項目顯示藥品名稱、劑型與對應病人

#### 3.4.4 新增藥品流程（Step Flow）

- Step 1：選擇所屬病人（未選病人時，下一步按鈕為 disabled）
- Step 2：填寫藥品資料（名稱必填、劑型必填、筆記選填）
- Step 3：確認資料並送出

#### 3.4.5 編輯藥品流程

- 所屬病人欄位為**唯讀**，不支援透過 API 變更（`PATCH /medications/{medication_id}` 不接受 `patient_id` 欄位）
- 可編輯欄位：藥品名稱（`name`）、劑型（`dosage_form`）、備註（`note`）
- 進入編輯頁後直接顯示現有資料供修改，所屬病人以唯讀方式呈現

#### 3.4.6 頁面狀態

| 狀態 | 行為 |
|------|------|
| 載入中 | 顯示載入指示器 |
| 空白狀態 | 顯示藥品列表為空或搜尋無結果的說明文字 |
| 正常狀態 | 顯示藥品列表，支援分頁 |
| 錯誤狀態 | 顯示錯誤訊息與重新整理按鈕 |

### 3.5 提醒排程管理

#### 3.5.1 功能描述

用戶可為特定病人的藥品建立提醒排程，設定服藥時間、劑量與重複規則。

#### 3.5.2 新增提醒流程（Step Flow）

- Step 1：選擇病人（支援搜尋與分頁；未選擇時下一步 disabled）
- Step 2：選擇藥品（支援搜尋；未選擇時下一步 disabled）
- Step 3：設定提醒規則（所有必填欄位驗證通過後才可前往確認）
- Step 4：確認資料並送出

#### 3.5.3 提醒規則欄位定義與驗證

| 欄位 | 說明 |
|------|------|
| 開始日期 | 必填，提醒排程的起始日期 |
| 服藥時間 | 必填，可設定多個時段（time slots） |
| 劑量 | 必填，數字，支援小數 |
| 劑量單位 | 必填，可選項目（微克、毫克、公克、毫升、公升、滴、錠、顆膠囊、顆、包、小包、個、噴、吸、片、次塗抹、顆栓劑、茶匙、湯匙、杯、單位、國際單位、毫當量、%、其他） |
| 排程類型 | 必填，分為一次性或重複（永不結束 / 到指定日期 / 指定次數） |

#### 3.5.4 重複排程規則與驗證

當排程類型為「重複」時，需額外設定：

| 欄位 | 說明 |
|------|------|
| 頻率單位 | 必填，日（day）、週（week）、月（month）、年（year）；切換為重複時預設為日 |
| 間隔 | 必填，正整數，最小值為 1，不可為小數；切換為重複時預設為 1 |
| 指定星期 | 僅頻率單位為「週」時顯示且必填，可多選星期幾（0–6） |
| 結束條件 | 三種：永不結束（never）、指定結束日期（until）、指定發生次數（counts） |
| 結束日期 | 結束條件為「until」時顯示且必填 |
| 發生次數 | 結束條件為「counts」時顯示且必填，正整數，最小值為 1 |

#### 3.5.5 排程類型切換的自動清空邏輯

- 切換回單次 → 清空頻率單位、間隔、星期、結束日期、發生次數
- 切換至永不結束（never）→ 清空結束日期與發生次數
- 切換至指定日期（until）→ 清空發生次數
- 切換至指定次數（counts）→ 清空結束日期

#### 3.5.6 編輯排程流程

- `PATCH /schedules/{schedule_id}` 為**全欄位替換**（非 partial update）；Required 欄位與新增時相同：`timezone`、`start_date`、`time_slots`、`amount`
- 所屬藥品（`medication_id`）與所屬病人（`patient_id`）為**唯讀**，不支援透過 API 變更
- 重複規則欄位的驗證邏輯與新增時完全相同（參見 3.5.4）

### 3.6 服藥紀錄

#### 3.6.1 功能描述

服藥紀錄分為兩個層級：提醒排程（Schedule）定義「應該何時吃藥」的規則；服藥歷史（History）記錄「那次事件實際上有沒有發生」的結果。兩者分開建模，確保歷史紀錄不受後續規則修改影響。

#### 3.6.2 設計概念：排程與歷史的分離

**排程（Schedule）** 是規則層，回答「這顆藥理論上何時該吃、如何重複、何時終止」。

**歷史（History）** 是結果層，回答「那次事件到底有沒有吃、實際吃了多少、是否有補充備註」。

系統在需要時由排程動態展開事件實例（event instance），只有實際被確認的事件才會落地為歷史紀錄永久保存。

#### 3.6.3 服藥紀錄的產生來源

| 來源 | 說明 |
|------|------|
| 快速確認（quickCheck） | 使用者在排程事件上點擊「已服用」，系統即時產生紀錄 |
| 手動補填（manual） | 使用者事後至歷史紀錄頁補填或修正服藥時間與資訊 |
| 系統補建（system） | 背景作業自動為已過期但未確認的事件補出 missed 紀錄 |

#### 3.6.4 服藥紀錄狀態

| 狀態 | 說明 |
|------|------|
| 待確認（pending） | 排程時間已到，尚未確認 |
| 已服用（taken） | 已確認服用 |
| 已逾時（missed） | 排程時間已過，系統標記為未服用 |

#### 3.6.5 使用情境

- 用戶在首頁或提醒觸發時，手動標記「已服用」（quickCheck）
- 進入歷史紀錄頁，依病人、日期、狀態等條件篩選紀錄
- 補填或編輯特定歷史服藥紀錄的內容（manual）

#### 3.6.6 篩選條件

- 單日日期（必選，預設為今日）
- 病人（選填）
- 藥品（選填，依選定病人連動）
- 紀錄狀態（選填：已服用 / 已逾時 / 待確認）

#### 3.6.7 歷史紀錄可編輯欄位

| 欄位 | 說明 |
|------|------|
| 實際服藥日期 | 必填 |
| 實際服藥時間 | 必填，格式為 HH:MM |
| 實際服用劑量 | 必填，數字，支援小數 |
| 身體感受 | 選填，1–5 分評分（很差 / 偏差 / 普通 / 良好 / 很好） |
| 備註 | 選填，最多 500 字 |

#### 3.6.8 歷史紀錄快照

歷史紀錄建立時會保留當下的快照，包含：

- **照護對象快照**：名稱
- **藥品快照**：藥品名稱、劑型
- **排程快照**：預定服藥時間、預定劑量與單位

即使後續藥品資料或排程被修改，歷史紀錄仍保留當時的真實內容，不會受到影響。

#### 3.6.9 頁面狀態

| 狀態 | 行為 |
|------|------|
| 載入中 | 顯示載入指示器 |
| 空白狀態 | 顯示「此日期沒有服藥紀錄」說明文字 |
| 正常狀態 | 顯示歷史紀錄列表，頂部顯示「全部 / 已服用 / 已逾時」統計數；支援篩選條件（日期、病人、藥品、狀態）與分頁 |
| 錯誤狀態 | 顯示錯誤訊息與重新整理按鈕 |

### 3.7 邀請與照護關係管理

#### 3.7.1 功能描述

用戶可透過邀請機制，邀請其他帳號加入照護關係，並設定對應的權限等級。邀請分為兩個方向：

- **邀請照護者（inviteCaregiver）**：邀請對方成為自己的照護者，讓對方可查看或管理自己的服藥資料
- **邀請被照護者（invitePatient）**：邀請對方成為被照護對象，讓自己可查看或管理對方的服藥資料

#### 3.7.2 邀請流程

- 發送者選擇邀請類型（照護者 / 被照護者）、填寫對象 Email 及指定權限（讀取 / 寫入 / 管理員）
- 被邀請者收到邀請後可接受或拒絕
- 接受後正式建立照護關係
- 邀請被拒絕或撤回後，可重新發送邀請

#### 3.7.3 邀請卡片顯示資訊

| 欄位 | 說明 |
|------|------|
| 邀請對象 / 邀請者 | 寄出方顯示對方姓名或 Email；收到方顯示邀請者姓名 |
| 邀請類型 | 邀請對方成為照護者 / 邀請對方成為被照護者 |
| 權限 | 讀取、寫入、或管理員 |
| 寄出時間 | 邀請發送的時間 |
| 狀態 | 等待回覆、已接受、已拒絕、已撤回 |
| 接受 / 拒絕 / 撤回時間 | 依狀態顯示對應的操作時間 |
| 到期時間 | 邀請的到期時間（若後端設有期限則顯示） |

#### 3.7.4 邀請管理功能

- 查看已發出的邀請（可依狀態篩選）
- 查看收到的邀請（可依狀態篩選）
- 撤回已發出的邀請（僅限等待回覆中）
- 接受 / 拒絕收到的邀請（僅限等待回覆中）

#### 3.7.5 頁面狀態

| 狀態 | 行為 |
|------|------|
| 空白狀態（已發出） | 顯示尚無發出邀請的說明文字，提供「新增邀請」捷徑 |
| 空白狀態（已收到） | 顯示尚無收到邀請的說明文字 |
| 操作後即時更新 | 接受、拒絕、撤回邀請後，列表即時反映最新狀態，不需手動重新整理 |
| 錯誤狀態 | 顯示錯誤訊息與重新整理按鈕 |

### 3.8 帳號設定

#### 3.8.1 功能描述

設定主頁作為帳號相關功能的統一入口，用戶可從此頁導向個人資料編輯、照護關係查看、照護邀請管理，以及執行登出。

#### 3.8.2 入口項目

| 項目 | 說明 |
|------|------|
| 編輯個人資料 | 導向個人資料編輯頁（3.9） |

## 4. 資料模型概覽

### 4.1 核心實體

| 實體 | 說明 |
|------|------|
| User | 系統帳號，可為照護者或被照護者 |
| UserSession | 登入 Session，管理 refresh token 生命週期 |
| Patient | 病人實體，可對應至 User 或純資料紀錄（如寵物） |
| Medication | 藥品，隸屬於特定 Patient |
| Schedule | 提醒排程規則層，關聯至 Medication |
| History | 服藥結果層，由 Schedule 事件確認後產生，含快照資訊 |
| CareRelationship | User 與 Patient 之間的正式照護關係，含權限等級 |
| CareInvitation | 邀請紀錄，狀態流轉：等待回覆 → 已接受 / 已拒絕 / 已撤回 |

### 4.2 主要關聯

- User 可擁有多個 Patient（直接建立）
- User 可透過 CareRelationship 關聯至其他 Patient
- Patient 可對應至一個 User 帳號（可為 null）
- Medication 隸屬於一個 Patient
- Schedule 關聯至一個 Medication
- History 由 Schedule 展開的事件確認後產生，保留當時快照
- CareInvitation 在雙方 User 之間流轉，接受後建立 CareRelationship

### 4.3 Schedule 與 History 的設計分層

這是系統最核心的資料設計概念：

- **Schedule（規則層）**：定義藥物何時應該被服用，可動態展開為事件實例
- **History（結果層）**：記錄每次事件實際發生的結果，一旦建立即為永久紀錄

這樣的分層讓排程規則可以隨時修改，而不影響已存在的歷史紀錄；同時避免預先產生大量未來事件資料。

## 5. 非功能需求

### 5.1 裝置支援

- 主要目標裝置：智慧型手機（iOS、Android）
- 實作方式：Vue.js（Nuxt）Web 應用程式，透過 WebView 嵌入行動端

### 5.2 權限控制

- 所有 API 呼叫需攜帶有效 Token
- 用戶僅能存取與其具有照護關係的病人資料
- 資料寫入操作需具備寫入（Write）或管理員（Admin）權限
- 邀請管理操作需具備管理員（Admin）權限

## 6. 已知限制與未來規劃

### 6.1 目前已知限制

- 不支援離線使用：所有操作需連線至後端 API
- 僅支援單一語言（中文）

### 6.2 未來規劃

| 功能 | 規劃方向 |
|------|---------|
| 離線使用 | 透過 Service Worker 快取關鍵資料，支援本地暫存與上線後同步機制 |
| 多語言支援 | 導入 i18n 框架，支援繁體中文以外的語系 |
